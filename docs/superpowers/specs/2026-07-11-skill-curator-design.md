# Skill Curator 公开包设计

日期：2026-07-11

## 目标

创建一个可公开下载的 `skill-curator` Skill，帮助用户盘点已安装的 AI Skills，识别重复、职责重叠和触发描述问题，并通过“只读诊断 -> 用户确认 -> 可逆处理 -> 永久删除”完成安全清理。

公开仓库是唯一源码。第一版不同时维护 `~/.agents/skills` 个人副本；用户需要在自己的 Codex 中使用时，可下载、复制或软链接公开目录。

## 第一版边界

包含：

- 扫描用户明确授权的 Skill 根目录。
- 解析 `SKILL.md` 的名称、description、路径、作用域、文件体积和修改时间。
- 检查同名、内容重复、软链接、缺失元数据和 description 质量。
- 根据证据分类为：保留、重写、合并候选、禁用候选、归档候选、删除候选。
- 为需要保留的 Skill 重写 description，并给出正向、负向触发测试语句。
- 用户确认后执行可逆的禁用或归档；永久删除必须再次确认。
- 提供单文件版本，供不支持目录式 Skill 的 AI 工具复制使用。

不包含：

- 不后台监控用户。
- 不声称仅凭文件修改时间能判断 Skill 是否被使用。
- 不在首次扫描中自动修改、移动或删除文件。
- 不删除系统、管理员或插件托管的 Skill。
- 不自动合并两个 Skill 的正文；只生成合并建议和候选结构。
- 不把“Skill 多”直接等同于“启动慢”。

## 包结构

```text
skills/skill-curator/

├── agents/openai.yaml

└── references/
    ├── description-guide.md
    └── safety-policy.md

skills/skill-curator.skill.md
```

同时更新：

- `skills/README.md`
- 仓库根目录 `README.md`

## 核心工作流

### 1. 发现与授权

先说明本次准备扫描的目录。默认建议常见位置，但只读取实际存在且在用户授权范围内的路径。遇到未知目录、权限错误或指向目录外的软链接时停止扩展扫描，并在报告中说明。

### 2. 只读盘点

`scripts/skill_inventory.py` 只负责确定性盘点，不做写操作。输出 JSON，包含：

- `path`
- `root`
- `scope`
- `name`
- `description`
- `size_bytes`
- `modified_at`
- `is_symlink`
- `symlink_target`
- `content_hash`
- `metadata_errors`

脚本不扫描 Skill 文件正文以外的个人内容，不读取密钥，不上传数据。

### 3. 诊断

AI 基于清单和必要的 `SKILL.md` 内容做判断：

- 同名：名称完全相同。
- 内容重复：`SKILL.md` 内容哈希相同。
- 可能重叠：目标任务、用户表达或触发词高度相似，但必须标注为模型判断，不当作确定事实。
- 触发不清：description 只有抽象能力、缺少使用场景、边界或真实用户表达。
- 使用状态：只有日志、用户陈述或其他可验证证据才能标记为“有使用证据”；文件修改时间只能代表文件变化。

### 4. 建议与确认

报告按以下动作分组：

1. 保留
2. 重写 description
3. 合并候选
4. 禁用候选
5. 归档候选
6. 删除候选

每项必须列出路径、证据、置信度、影响和建议动作。没有足够证据时选择保留或观察，不把“不确定”升级成删除建议。

### 5. 安全执行

- 修改前展示准确路径和 diff。
- 删除前优先建议禁用；Codex 可通过 `~/.codex/config.toml` 的 `[[skills.config]]` 设置 `enabled = false`。
- 归档时移动到用户确认的隔离目录，并生成可恢复清单。
- 永久删除必须在归档或禁用之后，由用户再次明确确认具体路径。
- 系统、管理员、插件缓存、包管理器托管目录只报告，不直接处理。
- Git 仓库中的文件修改前检查工作区状态，避免覆盖用户未提交更改。

## Description 优化规则

description 是触发选择器，不是功能宣传或工作流摘要。重写时：

- 前置最重要的使用场景和触发词。
- 写清对象、任务和边界。
- 覆盖用户真实会说的话及其同义表达。
- 必要时加入“不用于什么”，减少相邻 Skill 冲突。
- 保持简洁，避免“强大”“提升效率”等空泛表达。

每个重写结果附：

- 3 条应该触发的测试语句。
- 2 条不应该触发的测试语句。
- 与相邻 Skill 的边界说明。

## 输出契约

默认输出一份 Markdown 报告：

1. 扫描范围与限制
2. 数量和初始列表压力概览
3. 重复与冲突
4. Description 问题
5. 使用证据等级
6. 保留 / 重写 / 合并 / 禁用 / 归档 / 删除建议
7. 待用户确认的准确操作清单
8. 恢复方案

报告必须区分“确定事实”“启发式判断”和“用户待确认事项”。

## 错误处理

- YAML 解析失败：保留路径并记录错误，不跳过整个扫描。
- 权限不足：报告目录和错误，不请求扩大权限，除非用户要求继续。
- 软链接越界：只记录目标，不递归读取。
- 同名但不同内容：标记冲突，不自动判定哪一个正确。
- 找不到使用日志：标记使用状态未知，不用修改时间代替。
- 执行期间路径发生变化：停止该项操作，重新盘点。

## 验证标准

第一版交付必须通过：

- Skill 结构和 YAML 元数据验证。
- 盘点脚本测试：正常 Skill、缺失 metadata、重复内容、同名不同内容、软链接、无权限目录。
- 只读性测试：扫描前后文件树和内容哈希不变。
- 描述触发测试：清理、重复、误触发、漏触发场景应命中；创建普通新 Skill、普通文件清理场景不应命中。
- 安全检查：未经确认不得出现移动、禁用或删除动作。
- 单文件版与目录版的核心边界一致。

## 后续版本

如果用户确实需要按“过去 30/60/90 天是否调用”清理，应新增可选的本地使用记录器或 Hook。它需要单独说明日志来源、隐私范围和保留周期，不能在第一版里假装已有可靠使用数据。
